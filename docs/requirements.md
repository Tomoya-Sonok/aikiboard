## 1. プロダクト概要

### 1.1 コンセプト

AikiBoard は、合気道の道場・会などのコミュニティを運営する **管理者向けの道場管理＆コミュニケーションプラットフォーム** です。個人向け稽古日誌アプリ **AikiNote** の姉妹サービスとして、共通アカウント基盤のもとで連携しながら提供されます。

メタファーは「道場の掲示板（Board）」。AikiNote が個人の「ノート」であるのに対し、AikiBoard は道場に所属するメンバー全員が集まる「掲示板 / ダッシュボード」と位置づけます。

主軸は **管理者(オーナー・アドミン)の運営工数削減** です。メンバーのコミュニティ体験向上も AikiNote 連携によって同等に重視します。

### 1.2 ポジショニング

| | AikiNote | AikiBoard |
|---|---|---|
| 対象 | 個人の合気道家 | 道場・会の運営者＆所属メンバー |
| 主な価値 | 稽古日誌・個人記録 | 運営工数削減・コミュニティ運営 |
| 公開範囲 | 個人所有、任意で共有 | 道場単位のクローズドコミュニティ＋公開ページ |
| 主な課金者 | 個人ユーザー | 道場オーナー |
| 主要デバイス | スマホ(約 90%) | PC + スマホ両方(管理者は PC が主) |
| 関係性 | 姉妹サービス(共通アカウント) | 姉妹サービス(共通アカウント) |

### 1.3 LINE グループとの差別化

現状、多くの道場が LINE グループで代替しているため、以下の付加価値で差別化します。

- **稽古カレンダー**: 視覚的な月表示、公開ページとしても機能
- **出欠管理**: 参加表明＋集計、管理者向け集計ダッシュボード
- **AikiNote 連携**: 稽古日誌の引用共有、フィードのクロスポスト
- **道場ページ(公開プロフィール)**: 未ログインユーザー向けの集客チャネル
- **構造化されたコミュニケーション**: お知らせ / フィード / スレッド / アーカイブ
- **会計の見える化**: 月謝管理、収支可視化
- **アーカイブ**: 日々の稽古内容・議事録・演武会動画等を階層構造で長期保存

### 1.4 戦略的位置づけ(公開ページ = 集客チャネル)

公開ページは、既存メンバー向けの情報共有の延長ではなく、**「道場を探している人が発見・問い合わせできる集客チャネル」** として明確に位置づけます。

- SEO 対策された道場紹介・アクセス・指導者・稽古スケジュール(公開分)の表示
- 問い合わせフォーム / 見学申し込み導線の提供
- 既存の道場が公式ホームページを持たない場合、AikiBoard 公開ページを **「道場の Web 上の玄関口」** として機能させる

## 2. ターゲットユーザー

### 2.1 道場規模とプラン設計

メインターゲットは **小〜中規模**(5〜40 人程度)。将来的には全規模をカバーします。

プラン別のメンバー人数制限は以下の通り(料金・境界値は**仮**、運用後に調整予定):

| プラン | メンバー人数上限 | 月額(仮) | 年額(仮) |
|---|---|---|---|
| Free | 無制限(機能制限あり) | ¥0 | ¥0 |
| Mini | ≤ 20 人(仮) | ¥630 | ¥6,300 |
| Standard | 無制限(仮) | ¥980 | ¥9,800 |

初期は Mini / Standard の差別化は **人数差のみ**。機能差をつけたくなった時に `feature_flag` で柔軟に切り替えられる設計(詳細は 6.5)。

### 2.2 対応組織・流派

AikiNote の道場マスタ(`public.DojoStyleMaster`)に含まれる **すべての組織・流派** を初期から受け入れます。

- 合気会(Aikikai、既存 2,223 件の登録あり)
- 養神館(Yoshinkan)
- USAF(United States Aikido Federation)
- 心身統一合気道
- その他各流派・会派

AikiBoard オーナーは自分の道場が道場マスタに未登録の場合、AikiBoard 画面から新規追加できます(AikiNote 側にも反映される双方向書き込み。詳細は 5.2)。

### 2.3 ユーザーペルソナ

- **オーナー(道場長 / 会長)**
  - その組織のトップ 1 名
  - LINE や他の IT ツールに馴染みがない場合もある(その場合はアドミンに代理操作してもらう運用を想定)
  - プレミアムプランの課金責任者
- **アドミン(幹部 / 事務局メンバー)**
  - 運営補佐役。オーナー代理で管理操作を行う
  - オーナー譲渡・ボード削除・プレミアムプラン管理を除く管理権限を保持
  - 複数名登録可能
- **メンバー(一般道場生)**
  - 閲覧・参加表明・フィード投稿・スレッド返信が可能
  - 若年層から高齢層まで幅広い

### 2.4 アクセシビリティ方針

**世代・流派・国境を問わず、すべての合気道家が使いやすい** ことを目指します(AikiNote と同じ方針)。

- 高齢メンバーも想定: フォントサイズ・視認性・タップ領域は AikiNote と同等以上の配慮を予定
- 日英 2 言語対応(詳細は 8 章)
- 具体的な実装方針は AikiNote の規約をベースに採用(文字サイズスケーリング・WCAG AA 準拠・コントラスト確保等)

## 4. 機能要件

### 4.1 稽古カレンダー(最優先)

#### 4.1.1 カレンダー表示

- **月表示固定**(1 ヶ月ごとに前月・翌月へ切替)
- 稽古がある日付にはドット / バッジなどのインジケーターを表示
- 日付タップで**モーダル表示**: 日時、場所、担当指導者(任意)、備考、出欠状況
- **定期稽古(繰り返しイベント)** の設定対応

#### 4.1.2 公開設定

- ボード単位で **公開 / 非公開** フラグを管理者が設定
- **公開時**: 未ログインユーザーもカレンダーを閲覧可能(道場の公開ページとして機能、4.7 と連動)
- **非公開時**: ログイン済みメンバーのみ閲覧可能
- 想定では 95% 以上のボードが「公開」を選択

#### 4.1.3 出欠管理

- メンバーが各稽古イベントに対して **参加 / 不参加** を表明
- ログイン済みユーザーには、稽古イベントモーダル上で参加 / 不参加を表明したメンバーのプロフィール画像・氏名がひと目でわかる表示
- 出欠集計ビュー(管理者のみ閲覧可能なダッシュボード)

### 4.2 お知らせ配信

- 管理者(オーナー・アドミン)→ メンバーへの **一方向配信**
- メンバー全員への通知付き(通知手段は 4.9 参照)
- お知らせ一覧画面(既読 / 未読管理)
- リッチテキスト対応(見出し・太字・リンク程度)
- 管理者がお知らせ作成時に「メールでも通知する」フラグを ON にすることで、メンバーの登録メールにもメール送信

### 4.3 道場内フィード + スレッド

- メンバー・管理者の **双方** が自由に投稿可能
- AikiNote の「みんなで」機能の投稿一覧フィードと同じイメージ
- 投稿形式:
  - テキスト
  - 画像・動画
  - **AikiNote 稽古日誌の引用共有**(詳細は 5.3)
- 各投稿に対して **スレッド形式** で返信・議論可能(フラット 1 階層)
- 投稿時に「**AikiNote にも流す**」をチェックすると、AikiNote 道場アカウント経由で AikiNote 側のフィードにもクロスポストされる(詳細は 5.3)

### 4.4 アーカイブ(新規・MVP 必須)

> 道場内のフィード(双方向コミュニケーション)やお知らせとは別に、管理者が階層構造でページを作成して稽古の積み重ねを長期保存できる機能です。需要が高く、MVP に必須として組み込みます。

- **管理者のみ** が作成・編集可能。閲覧は全メンバー可。
- **階層構造のページ**(ツリー形式、親子階層で整理)。
- **用途例**:
  - 日ごとの稽古内容・議事録
  - 演武会・合宿の動画アーカイブ
  - 審査の記録
  - 指導者交代時の引き継ぎ資料
  - 道場の歴史・年表
- **添付**: テキスト / 画像 / 動画 / AikiNote 稽古日誌の引用
- **検索**: ページタイトル・本文のフリーワード検索
- **有料プラン限定** (Mini / Standard)

**詳細仕様(階層の深さ、添付サイズ上限、検索要件、ページテンプレート等)は別途詰める**(13 章「未決事項」に反映)。

### 4.5 メンバー管理

#### 4.5.1 メンバー一覧

- 全メンバーのリスト表示(ロール・参加日等)
- 各メンバーのプロフィール閲覧

#### 4.5.2 参加フロー

- **招待リンク**(URL / QR コード)経由での参加
- **AikiNote 上の道場データからの参加申請**(AikiNote の道場マスタと紐付け)
- 新規登録情報は AikiNote のユーザー作成フローに寄せる(AikiBoard のみ利用する場合でも AikiNote ユーザーが自動作成される。詳細は 5.1)

#### 4.5.3 退会・削除

- **メンバー自主退会**: メンバーが自分で道場から退会可能
- **管理者による削除・停止**: オーナー・アドミンがメンバーを削除 / 停止可能

### 4.6 アクティビティログ

- 管理者向けに、ボード内の操作履歴を記録・閲覧
- 例: 「○○さんが稽古スケジュールを追加」「○○さんがお知らせを投稿」「○○さんが参加表明」
- 管理者(オーナー・アドミン)のみ閲覧可能
- **有料プラン限定** (Mini / Standard)

### 4.7 道場ページ(公開プロフィール、集客用)

#### 4.7.1 MVP スコープ

- ロゴ、テーマカラーの設定(10 色プリセット、詳細は 9.4)
- 道場名・基本情報の表示(所在地、稽古場所、連絡先、指導者紹介、流派・組織等)
- 公開カレンダーの埋め込み(4.1.2 の公開設定に連動)
- 問い合わせフォーム / 見学申し込み導線
- SEO 対策(タイトル・ディスクリプション・構造化データ)
- URL: **`aiki-board.com/d/<slug>`**(パス型、10.4 参照)

#### 4.7.2 将来展望

- 道場 HP を持たない小規模道場向けに、**道場ホームページ兼会員向けページ** としての運用を訴求
- カスタマイズ可能な公開ページ(道場紹介・アクセス・指導者紹介等の追加ブロック)

- **有料プラン限定** (Mini / Standard)

### 4.8 会計の見える化(新規・MVP 必須)

> **ステータス管理 + 収支可視化** レベルの実装。オンライン決済(Stripe Connect 等による月謝のアプリ内決済)は **MVP 外** とします。

#### 4.8.1 月謝ステータス管理

- メンバーごとに月謝額を登録
- 支払い済み / 未払いのステータスを **手動で記録**
- 決済はオフライン(現金 / 銀行振込 / 手渡し)を前提

#### 4.8.2 支出管理

- 任意の支出項目(備品購入、合宿費、会場費等)を手入力
- 日付・項目・金額・備考・任意のカテゴリを記録

#### 4.8.3 収支可視化

- 月次 / 年次の収入・支出をグラフで表示
- オーナー・アドミンのみ閲覧可

- **有料プラン限定** (Mini / Standard)

**詳細仕様(月謝テンプレート、支出カテゴリの定義、グラフ種類、CSV エクスポート有無等)は別途詰める**(13 章「未決事項」に反映)。

### 4.9 通知(優先度: 最後)

- **プッシュ通知**: ネイティブアプリ(Phase 2)および PWA 向け
- **メール通知**: 管理者がお知らせ作成時に「メールでも通知する」フラグを ON にした場合にメール送信
- 通知対象:
  - お知らせの新規配信
  - 稽古スケジュールの変更・追加
  - (将来)フィード投稿やスレッド返信への通知

**MVP 内では基盤整備のみ、本実装は MVP の最後の工程** として着手します。

### 4.10 複数道場切り替え

- 1 ユーザーが **複数の道場(ボード)に所属可能**
- Slack のワークスペース切り替え UI のような切り替えインターフェース
- 各ボードごとに独立したロール・権限
- 通知は所属ボードごとに管理
- **有料プラン限定** (Mini / Standard) — Free プランは 1 道場まで

## 5. AikiNote 連携

### 5.1 共通アカウント基盤(SSO)

- AikiBoard と AikiNote は **別サービス** だが、**同一の Supabase Auth** で認証を統一します。
- `public.users` テーブルを両サービスで共有。
- AikiBoard 新規登録時に AikiNote のユーザーも同一トランザクション内で自動作成(AikiBoard のみ利用する場合でも AikiNote ユーザーが作成される)。
- 姉妹サービス間のシームレスな行き来を実現。
- 認証方式は AikiNote と同一(**Apple / Google ソーシャルログイン** を優先、メール / パスワードはセカンダリ)。

### 5.2 道場マスタ紐付け(双方向書き込み)

- AikiBoard のボード作成時に、AikiNote の道場マスタ(`public.DojoStyleMaster`)を **最低 1 件** 紐付ける必要がある。
- 1 ボード : N 道場マスタの紐付けを許容(将来、複数道場の合同ボード運営にも対応可能)。
- **1 道場が複数ボードを作成可能**(例: 子ども稽古 / 一般稽古、本部 / 支部など)。
- AikiBoard オーナーは自分の道場が道場マスタに未登録の場合、**AikiBoard 画面から新規追加** できる(AikiNote の道場マスタにも反映される**双方向書き込み**)。
- テーブル実装: `aikiboard.boards` と `public.DojoStyleMaster` を中間テーブル `aikiboard.board_dojo_masters` で N:M 紐付け(付録 A 参照)。

### 5.3 フィード連携(ユーザー選択式)

#### 5.3.1 AikiBoard → AikiNote クロスポスト

- AikiBoard のフィード投稿時に、投稿者が **「AikiNote にも流す」** をチェックできる(ユーザー選択式、投稿単位)。
- チェックされた投稿は、AikiNote 側のフィードに、AikiBoard の道場アカウント経由でクロスポストされる。
- AikiNote 側の公開範囲設定に従う。

#### 5.3.2 AikiNote → AikiBoard 引用共有

- AikiNote の稽古日誌を AikiBoard フィードに引用共有できる(SNS の引用ポスト同等)。
- **引用できる範囲**: AikiNote 投稿一覧に表示される投稿のみ(AikiNote 側の公開範囲設定に委ねる)。
- 公開範囲を絞って投稿一覧に表示させていないユーザーの投稿は、AikiBoard 側でも引用不可。

#### 5.3.3 実装アーキテクチャ

- **外部 API コールは不要**(同一 Supabase プロジェクト内のリレーションで完結)。
- `aikiboard.board_posts.synced_from_post_id`(`public.posts` への FK、nullable)で AikiNote 側の元投稿を参照。

### 5.4 DB・認証統合方針(概要)

- **DB**: 同一の Supabase プロジェクトを使用し、**スキーマ分離** で論理的に分ける
  - `public` スキーマ: AikiNote 既存テーブル + 両サービス共通テーブル(`users`, Auth 関連、`DojoStyleMaster` 等)
  - `aikiboard` スキーマ: AikiBoard 固有テーブルをすべて配置
- **RLS**(Row Level Security): `aikiboard` 全テーブルに適用
  - ボード所属メンバーのみデータアクセス可能 → `board_members` テーブルとの JOIN でポリシー定義
  - 公開設定 ON のボードは `events`(カレンダー)・`board_settings`(道場情報)のみ未認証ユーザーにも READ 許可
- **Worker**: AikiBoard 用 Cloudflare Worker を新規作成(AikiNote Worker とは別)、同一 Supabase に接続
- **共通テーブル変更の禁止**: `public.users` など既存テーブルは AikiBoard 側から変更しない(AikiNote の稼働を壊さないため、スキーマ変更が必要な場合は AikiNote 側と合意の上で実施)

詳細は 10 章「技術選定」参照。

## 6. 収益モデル

### 6.1 プラン構造

3 プラン構成:

- Mini / Standard の有料プランには、**30 日間の無料トライアル** を設ける。
- Free からのアップグレード時にトライアル開始。
- トライアル終了時に支払い情報未登録であれば Free プランにダウングレード。

### 6.4 支払い手段(Stripe + Stripe Invoice)

- **Stripe に統一**(AikiNote 同様)。
- **Web 決済のみ対応**(スマホ Web / PWA も可)。**ネイティブアプリ IAP は不要**(RevenueCat 不要)。
  - 理由: AikiBoard は管理者(PC が主)が支払いを行うため、ネイティブ IAP は不要。スマホユーザーも Web 決済で対応可能。
- **Stripe Invoice** にも対応:
  - オーナー個人 / 道場名義 の請求書発行を選択可能
  - 年度末の会計処理や組織の経費精算にも対応
- 年額プランは年一括支払い、月額プランは毎月自動更新。

### 6.5 feature_flag による拡張可能設計

現時点で Mini / Standard の機能差はないが、将来機能差をつけたくなったときに DB / コード変更なしで運用可能な設計とする。

**テーブル構成**(`aikiboard` スキーマ):

- `plans`: プラン定義(Stripe Product と対応、`id`, `stripe_product_id`, `code`, `member_limit`)
- `features`: 機能定義(`code`, `name`、例: `archive`, `public_page`, `accounting`, `activity_log`, `board_theme`, `multiple_boards`)
- `plan_features`: プラン × 機能の N:M 対応表
- `board_subscriptions`: ボードごとのサブスクリプション状態(`board_id`, `plan_id`, `stripe_subscription_id`, `status`, `trial_ends_at`, `current_period_end`)

**実装**:

- アプリ側で `hasFeature(boardId, 'archive')` ヘルパで実行時判定
- 機能の追加・プラン移動は `plan_features` テーブルの更新だけで完結(Stripe metadata 変更不要)

詳細は付録 A.6 参照。

## 9. デザイン方針

### 9.1 方向性

- **PC ファースト + レスポンシブ必須**
- 管理者は PC 利用が主、メンバーは PC / SP 両方を想定
- デザイン考案・実装は PC 先行、SP も同様に崩れない品質を担保
- 「管理ツールらしさ」を表現するため、UI 要素は **直線的・シャープ**
- AikiNote の「和風・柔らか」な雰囲気は色合いで踏襲しつつ、構造はより機能的に

### 9.2 デザインツール

- **Pencil**(Evolus Pencil)を採用
- 選定理由:
  - Figma に類似した UI と操作感
  - AI ツール操作に特化した設計
  - `.pen` ファイルは JSON に近い軽量テキスト形式
  - AI エージェントとの親和性が高く、AikiNote 含む AI 活用型開発フローに馴染む

### 9.3 AikiNote デザイントークン踏襲

- カラー: 墨 `#2C2C2C` / 銅 `#C4956A` / 和紙 `#F5F3EF` の 3 色基調
- モック `docs/draft/mock.tsx` の配色を参考
- 書体: Zen Old Mincho(ブランド)/ Noto Sans JP(機能 UI)
- アイコンは AikiNote と同じく Phosphor Icons を候補(リポジトリ初期化時に確定)
- 詳細は別途デザイナーとの打ち合わせで詰める

### 9.4 ボード別テーマカラー

- **MVP**: **10 色プリセット** から選択
- **MVP 外**: カスタムカラーピッカー(アクセシビリティ = コントラスト比担保の自動チェックが必要)

プリセット候補(デザイナー確定後に差し替え、13 章「未決事項」):
- 墨(デフォルト) / 銅 / 深緑 / 藍 / 臙脂 / 山吹 / 紫紺 / 鴇色 / 薄墨 / 鼠色

|---|---|

| DB 接続 | 同一 Supabase プロジェクト(SERVICE_ROLE_KEY) |

### 10.3 DB・Auth

- **DB**: Supabase(PostgreSQL)
- **Auth**: Supabase Auth(AikiNote と統一、1 プロジェクト 1 インスタンス)
- **プロジェクト**: AikiNote と **同一の Supabase プロジェクト** を使用、**スキーマ分離**
- **スキーマ**:
  - `public`: AikiNote 既存 + 共通テーブル(変更しない)
  - `aikiboard`: AikiBoard 固有テーブルすべて
- **RLS**: `aikiboard` 全テーブル適用(5.4 参照)
- **Backend → DB 接続**: `SUPABASE_SERVICE_ROLE_KEY`(RLS バイパス、server-side のみ)
- **Frontend → DB 接続**: `SUPABASE_ANON_KEY` + RLS で防御

詳細は付録 A「データモデル概要」参照。

### 10.4 ドメイン戦略

- **レジストラ**: **Cloudflare Register**(確定)
- **メインドメイン**: `aiki-board.com`(取得済み、2026-05-02、Cloudflare Register)
  - 本来希望の `aikiboard.com` は他社が取得済みのためハイフンつきの `aiki-board.com` で確定
- **サブドメイン構成**:
  - `aiki-board.com` — LP + 認証後のアプリ(Vercel ホスティング)
  - `api.aiki-board.com` — Cloudflare Workers API(`aikiboard-api`)
- **公開ページ URL**: **パス型 `aiki-board.com/d/<slug>`**
  - Next.js の動的ルートで SSR 実装が容易
  - SSL / 運用負荷が軽い(サブドメイン型 `<slug>.aiki-board.com` は却下)

### 10.5 リポジトリ戦略

- **別リポジトリ**(AikiNote と独立した GitHub リポジトリ)
- **Supabase 型定義**: 両リポジトリで各自生成(Supabase CLI で `public` + `aikiboard` を対象に)
- **共通 UI / ユーティリティ**: 初期は **コピー許容**。肥大化したら別パッケージ化を検討
- **CLAUDE.md / `.agent/` ルール**: 両リポジトリで類似構造を維持(AI エージェント開発フローの共通化はここで実現)
- npm パッケージ化は個人開発では過剰なため当面採用しない

## 付録 B: 用語集

| 用語 | 意味 |
|---|---|
| **ボード(Board)** | AikiBoard における道場の単位。1 オーナー 1 ボード作成時点、メンバーは複数ボード切替で複数所属可能 |
| **オーナー** | ボードの最上位管理者。プラン課金責任者。1 ボード 1 名 |
| **アドミン** | ボードの管理補助者。複数名登録可 |
| **メンバー** | ボードに所属する一般道場生 |
| **道場マスタ** | AikiNote 側の `DojoStyleMaster` テーブル。既存 2,223 件の合気会道場等を登録済 |
| **クロスポスト** | AikiBoard のフィード投稿を、ユーザー選択で AikiNote 側にも流す機能 |
| **引用共有** | AikiNote の稽古日誌を、AikiBoard フィードに引用表示する機能(SNS の引用ポスト同等) |
| **feature_flag** | プラン別機能制御の仕組み。`plan_features` テーブルで N:M 管理 |
| **公開ページ** | 未ログインユーザーもアクセスできるボードの集客ページ。`aiki-board.com/d/<slug>` |
| **スキーマ分離** | 同一 Supabase プロジェクト内で `public`(AikiNote)と `aikiboard`(AikiBoard)をスキーマで分ける方針 |
| **SSO** | シングルサインオン。同一 Supabase Auth を両サービスで共有 |
| **アーカイブ** | 管理者が階層構造のページで稽古内容・議事録・演武会動画等を蓄積する機能 |
| **RLS** | Row Level Security。Supabase(PostgreSQL)の行単位アクセス制御 |
| **Stripe Invoice** | Stripe の請求書発行機能。個人・法人名義での請求書発行に対応 |
